3 approches possibles

Fusion Ecosystem + améliorations Analytics

A

Réorganisation légère ★ recommandée

Ecosystem : fusionner use-cases + guide en une seule page /ecosystem avec sticky anchor nav (2 sections : Who Builds → Run a Node). Les anciennes routes redirigent vers la nouvelle. Minimum de nouveau code, on réutilise les composants existants.

Analytics : toggle pie/bar sur ProviderComparison · trend chart étendu à toutes les chains · widget "Top OVH Operators" (acteurs les + présents, cliquable vers leur fiche node).

Pour

  • Rapide à implémenter
  • Réutilise 95% du code existant
  • Impact UX immédiat

Contre

  • Layout Who Builds = même grille qu'aujourd'hui
B

Redesign + Ecosystem hub

Ecosystem : idem A, mais "Who Builds" gagne un filtre par chain (afficher seulement les acteurs Avalanche, ou multi-chain). "Run a Node" devient un accordéon step-by-step avec des blocs de code copiables.

Analytics : idem A + la géo distribution passe en carte monde interactive (react-simple-maps) plutôt que barres horizontales.

Pour

  • UX plus riche
  • Filtre utile quand tu ajoutes des chains

Contre

  • +2–3 jours de dev
  • Filtre peu utile si peu de case studies par chain
C

Ecosystem = social proof uniquement, guide absorbé dans Dashboard

Ecosystem : la page ne garde que "Who Builds" (case studies + verified residents). Le guide technique (hardware, deploy) disparaît en tant que page et est transformé en side panel / drawer accessible depuis le Dashboard.

Analytics : idem A.

Pour

  • Ecosystem = page propre, 100% focalisée social proof

Contre

  • Le guide technique dans un drawer = moins visible
  • Casse la mémoire de navigation des utilisateurs actuels
